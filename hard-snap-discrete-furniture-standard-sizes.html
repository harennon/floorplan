Hard-snap discrete furniture to standard sizes

Some furniture is genuinely discrete: beds come in 6 mattress sizes, ranges / fridges / washers in fixed 24 / 30 / 36″ widths. The catalog already carries these as presets, but resize still free-scales within the per-axis bounds — so a bed can land at 1.20 m wide, between Full (1.37) and nothing real.

These three options each answer one question: when a piece is flagged discrete:true, how does resizing feel? They are ordered from most-automatic to most-explicit. The stages below are static illustrations of the key states.

Option A · Magnetic rungs

Resize magnetically snaps to the nearest preset

Dragging a resize handle (or typing a value) is continuous, but on release it hard-snaps to the nearest preset instead of clamping to the continuous range. During the drag, faint "rung" outlines mark every valid size; the one you're about to land on lights up green and its name shows in the cursor snap-tag. Reuses the existing wall-flush / alignment snap-tag vocabulary — nothing new to learn.

Best for: feels native to the tool. No extra clicks. The chip stays a plain measurement. Tradeoff: you can't see the whole size ladder at once, and a fine drag can feel "sticky".

Option B · Size chip → picker

The dimension chip becomes a size picker

For discrete types the chip stops being a free-text field. It shows the current preset name next to the measurement (Queen · 1.52 m ▾). Tapping it opens a small popover listing every real size with its dimensions; the current one is ticked. This makes the whole ladder visible and turns resize into a recognition task rather than "guess the number". A dragged handle still hard-snaps (as Option A) so both entry points agree.

Best for: discoverability — a first-timer sees "oh, beds come in these sizes". Mobile-friendly (tap targets, no fiddly drag). Tradeoff: the chip is wider and behaves differently from the free-scale chips on non-discrete furniture.

Option C · Snap-on-commit

Keep the text field — snap the entered number, tell the user

The smallest change. The inline text input stays exactly as it is today, so nothing about the interaction moves. When you type a value and hit Enter, the number snaps to the nearest preset and a toast confirms what happened ("Snapped to Queen · 1.52 m") with a one-tap Undo. The input gets a subtle teal ring + preset suggestions (a native datalist) while you type, hinting the field is discrete. The chip afterwards can optionally show the preset name.

Best for: minimal surface area / fastest to ship — reuses symbolDimEntry.js verbatim, just swaps clamp for snap in resizeSymbol. Tradeoff: least discoverable (you only learn sizes are discrete after a snap surprises you).

Which one?

All three share the same core rule (discrete:true types snap resize to the nearest preset instead of clamping). They differ only in how visible that rule is at the moment of resize. They can also stack — B and C are complementary, and A's magnetic drag pairs with either.

MCP parityAll three: resize_symbol / place_symbol snap server-side identically; the result reports snappedTo preset name.

Recommendation: ship C as the baseline behavior (the snap rule everyone gets) and layer B on top for discoverability — the discrete chip becomes a picker, and typing still works. Option A's magnetic drag is a nice-to-have that can follow once handle-drag resize exists. Under the hood all three need the same two things the issue calls for: a discrete:true flag on the relevant CATALOG entries (bed, fridge, stove, washer, and any other genuinely-standardized type), and a snapToPreset(type, dim, metres) helper that resizeSymbol and the MCP mutators call instead of clampDim for discrete types.
