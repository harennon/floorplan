Catalog: desktop monitor + gaming chair

Two new furniture types (LLD 98, follow-up from LLD 72a). The frontend surface is small and generic: two dock buttons + two canvas glyphs. Everything else (per-axis bounds, presets, resize, clearance, export, share) flows through the existing catalog-driven paths — no per-type code. The real decision below is which category tab the two pieces live under.

The decision

Which category tab do the two pieces live under?

The dock groups items by category tab (Openings / Living / Kitchen / Bedroom / Bath / Outdoor). Both new pieces read as a desk/gaming setup, but there is no "gaming" tab today. Two options — the difference is entirely the category field in CATALOG and where the two buttons sit.

Option A Recommended

Fold both into the existing Living tab, beside desk and tv — category:"living".

  • Zero new UI structure — matches the LLD 45 "extend, don't restructure" decision.
  • A desk battlestation is already a Living-tab scene (desk + tv live there).
  • Keeps the tab count at 6; no empty/sparse tab.
  • "Gaming" isn't a literal label — pieces are general desk furniture, which is fine.

Option B Alternative

Add a new Gaming tab holding monitor + gaming-chair — category:"gaming", a 7th tab.

  • Explicit home; discoverable for the "battlestation" use case.
  • A 2-item tab looks sparse next to Living's 9 items.
  • New SymCategory value + tab button + tabpanel + router wiring — more surface, more risk.
  • Scope creep vs. LLD 72a, which explicitly defers the gaming context, not a UI tab.

The data (proposed)

Catalog entries & the [GEO] rule they unlock

Per-axis bounds + presets, exactly like the rest of CATALOG. With a real desk-scale monitor, the gaming-room skill can place a genuine battlestation and check seat-to-screen viewing distance ~0.5–0.75 m as a [GEO] rule instead of approximating via desk depth (LLD 72a "Catalog gaps"). Final numbers are the architect's to lock in the LLD; these are grounded starting points.
