Kitchen counter / countertop + island

Catalog addition mockup · LLD 105 · follow-up from LLD 72a “Catalog gaps”. New catalog types let the interior-design skill (and a future check_layout) measure landing zones instead of describing them in prose.

These are throwaway design artifacts, not production code.

The decision: one type or two?

Counter (against-wall prep run) and island (free-standing prep block) are the two catalog gaps LLD 72a names. They differ only in default footprint and where they seat — both are plain rectangular furniture in the kitchen category. The question is whether to ship them as one flexible type or two distinct dock items.

Option A — two types recommendedOption B — one counter type
Dock items Two: Counter and Island, side by side in the Kitchen tab. One: Counter; a very wide/deep preset stands in for an island.
Skill / check_layout Can reason “counter run vs island” directly by type — matches kitchen.md’s distinct “run” vs “island” aisle language. Must infer island-ness from size — fragile heuristic; no clean predicate.
Presets Counter: standard 61 cm-deep runs by length. Island: 61–122 cm deep blocks. Each list stays short and real. One mixed list mingling shallow runs and deep blocks — harder to scan.
Glyph read Counter = seam + backsplash edge; Island = seam + rounded overhang. Distinct at a glance. One glyph must serve both roles.
Cost Two CATALOG entries + two glyph branches + two dock buttons. Follows the exact post-#98 pattern (bounds + presets, non-circular, non-discrete). Marginally less markup; pushes complexity onto the skill and the user.

Recommendation: Option A (two types). It is the small-surface-area choice that actually unblocks a check_layout evaluator — the evaluator can key landing-zone rules off type === "counter" / "island" rather than a size guess. The rest of this mockup shows Option A; the dock section also previews Option B so you can compare.

1 · Palette / dock placement

New items live in the existing Kitchen tab of the symbol dock, appended after the four appliances. Same .dock-item button (18×18 inline SVG glyph + label) as every other piece.

“new” hint fades after first session (decorative only in this mockup)

2 · Glyphs & catalog geometry

Rendered the way symbolRender._renderInterior would draw them: a body polygon plus a type-specific interior glyph. Counter reads as a run (long seam + thin backsplash strip on the wall edge); island reads as a free block (seam + rounded overhang lip, no backsplash).

Per-axis bounds & presets follow the post-#98 catalog contract (see src/js/symbols.js). Both are ordinary furniture: not circular, not discrete (a counter is cut-to-fit, so free resize within bounds + optional preset chips — no hard snap). anchorable like appliances so a counter seats flush to a wall run.

3 · Inspector — presets & color

When a counter or island is selected, the inspector shows preset chips (#insp-presets) and the kitchen color swatch strip (appliance + wood + neutral, per palette.js). Counters are typically stone/wood, so wood + neutral swatches lead.

4 · The payoff — measurable landing zones

This is why the type exists. Today the skill can only say “leave ~0.38 m of counter beside the fridge”. With a real counter object, clearance annotations (the existing overlay in clearanceRender.js) measure the actual gap between appliance and counter — and a future check_layout can pass/fail each NKBA landing rule.
