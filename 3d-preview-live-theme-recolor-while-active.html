3D preview — live theme recolor while preview is active

Issue #132 · follow-up to LLD 130 · the 3D scene stands in for the three.js canvas; colors are CSS vars so a toggle triggers a real recolor.

What this issue is really about. The engine change is tiny: rebuild _planGroup on onThemeChange() while preview.isActive(). The design decision is the prerequisite — should the theme toggle even be reachable during preview, and if so, how does the recolor read? The options below answer both.

Option A — Keep the theme toggle out of preview (status quo) no engine change

The actions cluster (with the ☀/☽ toggle) stays hidden during preview, exactly as today. The theme can only change on the next preview entry, where colors already re-derive correctly. This is the current LLD 130 Edge Case 10 behavior — shown as the baseline to react against. No live-recolor path; #132 stays deferred.

Option B — In-scene theme control + soft recolor recommended

Expose one minimal ☀/☽ button pinned to the 3D stage (not the full actions cluster — that stays hidden to preserve read-only). Toggling it recolors the live scene with a brief cross-fade (~0.4s) so the change reads as intentional rather than a jarring flash. This is the case that makes #132's engine work worthwhile: rebuild _planGroup on onThemeChange, but ease materials to the new palette instead of a raw dispose flash.

Option C — In-scene control + hard rebuild snap simplest engine

Same pinned control as B, but the recolor is a literal dispose + rebuild in one frame: colors snap with a momentary tear-down dim (the honest depiction of what the minimal onThemeChange → _disposePlanGroup(); _buildPlanGroup() does with no easing). Cheapest to build and closest to the issue's literal scope, but the flash can read as a glitch on large plans.

Recommendation: Option B. If the theme becomes reachable during preview at all, a single pinned control keeps the read-only contract (no dock/inspector re-exposed) while the soft cross-fade makes the recolor feel like a deliberate view change. Option A remains valid if we decide the theme should simply stay unreachable (then #132 stays closed as deferred). Option C is the fallback if easing three.js materials proves fiddly and we accept the snap.
