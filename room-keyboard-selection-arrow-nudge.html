LLD 63 follow-up · Issue #82

Room selection & arrow-key nudge

Rooms already move by pointer-drag (LLD 63). This adds the keyboard nudge path that symbols got in LLD 54 — so a selected room can be moved with the arrow keys, at the same step, with the same one-undo-per-burst behavior.

User feedback carried in: “I'd prefer if the room selection works the same way as object selection — select via click, move via arrow keys.” So there is no Tab-cycling: selection stays pointer-click (identical to symbols), and only the nudge is added. The directions below differ only in the feedback / discoverability layer that LLD 63 left open.

Direction A — Silent parity with symbols

The room nudge behaves exactly like the symbol nudge: click a room to select (dashed gold outline), press an arrow key to move by one grid step, Shift+arrow for a coarse 4× step. No new chrome at all — the moving outline is its own feedback, just as a moving symbol is. Lowest cost, most consistent, invisible until used.

Interaction model

  • Select: click/tap room interior (unchanged from LLD 63)
  • Nudge: ↑↓←→ = one grid step
  • Coarse: Shift+arrow = 4× step
  • Undo: one burst (≤400 ms gap) = one undo step
  • Carried furniture moves with the room (LLD 63 rule)

Why this direction

Perfect symmetry with symbols means zero new mental model and near-zero code: main.js already owns the arrow-key handler for symbols. We only branch it to route to a new roomTool.nudgeSelected when a room is selected.

Risk: no explicit signal that keyboard nudge exists — discovered by trying, or via the ? shortcuts overlay (which we'd extend).

Direction B — Nudge readout badge

Same click-select + arrow-nudge as A, but during an active nudge burst a small live delta badge tracks the room, showing how far it has moved (e.g. +0.30 m →). It fades out shortly after the burst commits. Gives a numeric anchor for “does my couch fit” nudging without any persistent chrome.

The badge accumulates the burst delta and fades ~1 s after you stop. Coarse (Shift) jumps show as 4× steps.

What it adds over A

  • A transient badge pinned to the room during a burst
  • Reads signed distance from the burst start, in current units
  • Fades out after the debounce commit (~400 ms + fade)
  • Nothing persistent; silent when idle

Trade-offs

Pro: supports the “fit” use-case — you can nudge a room a precise 0.30 m and see it. Reinforces that arrows moved something.

Con: a little more render code (a positioned overlay that tracks the room), and symbols don't have this today — so it introduces a small asymmetry unless we later add the same badge to symbol nudge.

Direction C — Discoverable (hint + focus ring)

Same click-select + arrow-nudge, plus two discoverability cues: (1) a faint teal focus ring just outside the selection outline signals “this is keyboard-active — arrows will move it,” and (2) a one-time coach hint the first time a room is selected: “Arrow keys nudge · Shift = bigger step.” Highest discoverability, most chrome.

The coach hint appears once per session on first room-select, then auto-dismisses. The focus ring stays while the room is selected.

What it adds over A

  • Focus ring: dotted teal, outside the gold outline — reads as “keyboard target”
  • Coach hint: one-time toast on first room selection
  • Session-only flag (never persisted) tracks “hint shown”

Trade-offs

Pro: most accessible — the keyboard capability announces itself, aiding the stated a11y goal. Best for first-time users.

Con: most chrome and the biggest departure from the minimal-chrome principle and from symbol parity (symbols show no ring / hint). Two extra visual elements to build, theme, and test across light/dark.

Recommendation: ship Direction A for v1 — it is the literal reading of the user's feedback (“works the same way as object selection”), the cheapest to build, and keeps symbol/room behavior symmetric. The ? shortcuts overlay is extended with a “Nudge selected (symbol or room)” row so it is still discoverable. Direction B's readout badge is a strong later add-on for the “does it fit” wedge (ideally added to symbols too, for symmetry). Direction C is the fallback if QA finds the keyboard path too hidden.
